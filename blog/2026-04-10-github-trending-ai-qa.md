今天早上的 GitHub Trending AI 项目研究，有一个特别强烈的感受：**AI 不再只是“模型能力”竞赛，而是在“工程化能力”上全面开卷**。对测开来说，这意味着两个方向同时变得更清晰：

- 一方面，AI Agent/基础设施的形态正在快速收敛，可测试的边界更明确。
- 另一方面，这些项目沉淀出来的工程化设计（可观测、可回放、可评测、可治理）会反过来“降维”提升 QA 的效率与确定性。

本文按测开视角，把研究要点整理成 3 个模块，并给出基于 `Ginkgo` / `Playwright` 的落地建议。

> 说明：本文聚焦“可复用的方法论与工程化抓手”，不强行罗列具体项目清单；你可以把它当成每日 Trending 研究的固定写作模板。

## AI Agent 与基础设施的新趋势

从近期 Trending 的形态看，AI 相关项目大体在向几条主线聚合，且每一条都天然对应 QA 的可测点：

### 1) Agent 从“聊天”走向“执行”

典型结构不再是单轮问答，而是更像一个可编排的执行系统：

- **规划（Planner）/执行（Executor）分离**：把“想做什么”和“怎么做”拆开，便于插入规则、限权、审计。
- **工具调用（Tool/Function Calling）规范化**：输出结构化 JSON、可验证 schema、可重试的错误码。
- **多 Agent 协作**：角色分工（检索/分析/执行/审计），但更依赖“上下文管理”和“冲突仲裁”。

测开关注点：

- 工具调用是否满足 `JSON Schema`（结构、必填、枚举、约束）。
- 规划是否可复现（同输入是否有稳定的步骤集合、是否可控随机性）。
- Agent 之间的消息边界是否可观测（是否能定位谁造成了错误决策）。

### 2) 基础设施从“能跑”走向“可治理”

基础设施项目的热点不只是向量检索/RAG，而是更完整的“可运营套件”：

- **评测/回归（Eval / Regression Harness）**：把模型升级、Prompt 改动、知识库变更纳入回归体系。
- **可观测（Logs/Metrics/Traces）**：对一次 Agent 执行给出 end-to-end 的链路追踪。
- **安全与防护（Guardrails）**：输入输出审查、提示注入防护、敏感信息脱敏、权限控制。
- **成本与性能（Latency/Token/Cache）**：缓存、并发、限流、降级策略逐步标配。

测开关注点：

- 评测集是否具备“可解释失败”的结果（失败案例能否直达根因：检索错、工具错、推理错、数据错）。
- 链路是否覆盖关键 span（检索、rerank、调用外部 API、生成、解析、写入）。
- 降级/熔断/重试是否可验证（以及是否会放大不稳定性）。

### 3) 数据与知识从“堆料”走向“版本化资产”

越来越多项目在强调：Prompt、工具定义、知识库、评测集都应该像代码一样被版本化。

测开关注点：

- Prompt/知识库变更是否触发回归（像 `go test`/`playwright test` 一样自动化）。
- 线上事故是否能“回放同一上下文”（确定性复现是质量治理的核心杠杆）。

## 质量保障的降维打击

“降维打击”不是让 QA 被替代，而是：**工程化设计把不确定的 AI 行为变成可测、可控、可回归的对象**。

下面是几条对 QA 直接增益的工程化抓手，你可以把它们当成 AI 项目质量体系的“骨架”。

### 1) 把不可测的自由文本，约束成可验证的接口

- 统一对外输出为结构化对象（JSON），对齐 `schema + 示例 + 反例`。
- 解析失败要有明确错误码与可重试语义（别把失败吞进“模型又抽风了”）。

落到测试上：

- 增加 **schema 合规性测试**（单测级别、快、稳定）。
- 增加 **关键字段语义约束**（枚举/范围/一致性，比如 `severity`、`confidence`、`action`）。

### 2) 把一次 Agent 执行，拆成“可断言的阶段”

典型可断言阶段：

1. 输入归一化（脱敏、补全、校验）
2. 检索（召回/重排）
3. 规划（步骤、工具选择）
4. 执行（工具调用、外部依赖）
5. 汇总输出（结构化、引用来源、置信度）

落到测试上：

- 失败要能落到阶段：检索错/规划错/执行错/汇总错。
- 每个阶段给出最小可观测指标：命中率、延迟、token、重试次数、错误分布。

### 3) 把“模型/Prompt/知识”的变化，纳入回归与发布治理

- Prompt、工具定义、知识库、评测集：都用版本号/变更记录管理。
- 发布策略像服务发布：灰度、回滚、金丝雀、对照组。

落到测试上：

- 用一套固定评测集跑出“变更前/后差异”，把主观感受变成数据。
- 对高风险场景加红线：例如 PII 泄露、越权访问、错误执行不可逆动作。

### 4) 把“线上问题”变成“可复现用例”

- 记录执行上下文（输入、检索结果摘要、工具请求/响应、关键中间产物）。
- 支持 record/replay（至少做到“同一上下文能重跑，输出可比较”）。

落到测试上：

- 引入“线上问题回归库”，每次修复/Prompt 调整都必须跑过。
- 对不稳定用例做隔离与治理（flaky 不是“忍了”，要可定位、可压制、可修复）。

## 可落地行动指南

下面给一份“下周可执行清单”，按你现有 `Ginkgo` / `Playwright` 技术栈设计，目标是在不重构系统的前提下，把 AI Agent 的质量保障快速跑起来。

### A. 用 Ginkgo（Go）把后端 Agent/工具链测稳

**1) 建一个最小可用的测试分层**

- `contract`：schema/错误码/幂等性（最快、最稳定）
- `integration`：工具调用 + 外部依赖 stub（可控环境）
- `e2e`：端到端执行链路（少量关键路径）

**2) 关键测试用例建议**

- 输出结构校验：对每个核心能力定义 JSON schema，失败时输出差异。
- 工具路由正确性：相同输入在不同“工具可用集”下，选择应符合预期。
- 重试与超时：对外部 API 注入超时/5xx，验证重试次数、退避策略、最终错误码。
- 权限边界：RBAC/租户隔离/只读与写入工具的限权（越权必须硬失败）。

**3) 可操作的工程实现（建议）**

- 用 `httptest`/mock server stub 掉 LLM 与外部依赖，让测试确定性更强。
- 用 golden file（黄金文件）固化关键输出（建议只固化结构与关键字段，避免全量文本导致噪声）。

### B. 用 Playwright 把“用户路径 + Agent 交互”测透

**1) 优先覆盖三条高 ROI 场景**

- Agent 在 UI 中的关键任务流（例如：输入 → 规划展示 → 执行 → 结果落库/导出）。
- 失败兜底体验（超时、无权限、外部依赖失败时的提示与恢复动作）。
- 可观测证据（trace/log link）是否在 UI 中可追溯。

**2) 稳定性策略**

- 对不可控内容做“结构断言”：断言状态机、关键字段、引用来源，而不是全文一致。
- 使用 `trace`/`video` 作为失败复盘资产，并把用例按 flaky 策略隔离。

**3) 结合接口测试提升端到端确定性**

- 用 `APIRequestContext` 预置数据（seed），避免用 UI 做所有前置。
- 必要时对网络层做 mock（例如：固定检索结果、固定工具返回），将 UI 验证聚焦在交互与编排正确性。
